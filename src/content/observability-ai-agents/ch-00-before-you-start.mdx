## Who this series is for

This series is for engineers who already have agents running, or are close to putting them in production, and are starting to feel the visibility gap. The agent calls a model, chooses tools, reads context, writes output, maybe loops, maybe escalates, and the dashboard still mostly shows latency and a status code.

You do not need to be an observability specialist to follow along, but the examples assume comfort with backend development, structured logging, APIs, and the idea of distributed tracing. The applied chapters use Python, LangGraph, OpenTelemetry, Docker Compose, and Langfuse.

## What this series assumes

You already understand the basics of LLM-powered applications:

- A model call has inputs, outputs, token usage, latency, and a finish reason.
- An agent can call tools, retrieve documents, keep short-term state, and run multiple steps before answering.
- Production systems need security controls, budget controls, and incident response, not just a good demo.

Terms like trace, span, attribute, PII, retrieval, guardrail, and LLM-as-judge appear throughout the series. The chapters introduce them in context, but the pace assumes you are already comfortable reading technical material.

## Required tools for the hands-on chapters

The first eight chapters are conceptual and design-focused. The implementation starts in Chapter 9.

For the hands-on part, install:

```sh
python --version
# Expected: Python 3.11 or higher

docker --version
# Expected: Docker installed and running

docker compose version
# Expected: Docker Compose v2
```

You will also need access to an LLM provider. The examples are written so the provider can be swapped, but you should have one model available before starting the implementation chapters.

## The example stack

The hands-on path uses:

| Layer | Tool | Why |
|---|---|---|
| Agent framework | LangGraph | Explicit state and control flow make agent traces easier to reason about. |
| Telemetry API | OpenTelemetry | It gives us vendor-neutral terms for traces, spans, and attributes. |
| Observability backend | Langfuse | It is designed for LLM applications and can be self-hosted for the series. |
| Local environment | Docker Compose | It keeps the observability backend reproducible. |

Langfuse is the backend used in the examples. The telemetry structure stays portable because the instrumentation goes through OpenTelemetry.

## Questions this series will help you answer

- Which agents or workflows do you want to observe?
- Which tools can those agents call?
- Which data can appear in user input, tool results, retrieval context, and model output?
- Who can inspect traces?
- What is the maximum acceptable cost per request, session, or day?
